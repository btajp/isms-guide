# Requirements: 仮想組織の設定

## Overview

全93管理策に対応するテンプレートを作成するにあたり、物理的セキュリティなど組織固有の内容が多い管理策については、仮想的な組織を設定する必要がある。この仮想組織は、テンプレート内のサンプルデータや記載例の基盤となり、フォークユーザーが自組織に置き換える際の参考となる。

## User Stories

### US-1: テンプレート作成者として一貫した組織設定を参照したい

**As** ISMS Guide のテンプレート作成者
**I want** 一貫した仮想組織の設定を参照したい
**So that** 各テンプレートで矛盾のないサンプルデータを作成できる

#### Acceptance Criteria

1. **AC-1.1**: WHEN テンプレート作成者が仮想組織ドキュメントを参照する THE SYSTEM SHALL 組織の基本情報（名称、規模、業種）を表示する
2. **AC-1.2**: WHEN テンプレート作成者が仮想組織ドキュメントを参照する THE SYSTEM SHALL 組織構造（部門、役職、従業員構成）を表示する
3. **AC-1.3**: WHEN テンプレート作成者が仮想組織ドキュメントを参照する THE SYSTEM SHALL 拠点情報（本社、支社、データセンター）を表示する
4. **AC-1.4**: WHEN テンプレート作成者が仮想組織ドキュメントを参照する THE SYSTEM SHALL 扱う情報資産の種類を表示する

### US-2: フォークユーザーとして仮想組織を自組織に置き換えたい

**As** リポジトリをフォークして自社 ISMS を構築するユーザー
**I want** 仮想組織の設定を自組織の情報に置き換えたい
**So that** テンプレートを自社向けにカスタマイズできる

#### Acceptance Criteria

1. **AC-2.1**: WHEN ユーザーが仮想組織ドキュメントを閲覧する THE SYSTEM SHALL 置き換えるべき項目が明確にマークされている
2. **AC-2.2**: WHEN ユーザーが仮想組織ドキュメントを閲覧する THE SYSTEM SHALL 各項目の説明と記載例が含まれている
3. **AC-2.3**: WHEN ユーザーが仮想組織ドキュメントを閲覧する THE SYSTEM SHALL 組織規模別のカスタマイズ指針が含まれている

### US-3: サイト閲覧者として仮想組織の設定を理解したい

**As** ISMS Guide を参照する組織の担当者
**I want** テンプレートで使用されている仮想組織の設定を理解したい
**So that** サンプルデータの背景を把握してテンプレートを正しく解釈できる

#### Acceptance Criteria

1. **AC-3.1**: WHEN ユーザーがテンプレートページを閲覧する THE SYSTEM SHALL 仮想組織ドキュメントへのリンクを表示する
2. **AC-3.2**: WHEN ユーザーが仮想組織ドキュメントを閲覧する THE SYSTEM SHALL 仮想組織がサンプル目的であることを明示する

## Functional Requirements

### FR-1: 仮想組織の基本情報

以下の基本情報を定義する：

| 項目 | 内容 |
|------|------|
| 組織名 | 株式会社テクノソリューションズ（仮称） |
| 業種 | IT サービス業（SaaS プロバイダー） |
| 従業員数 | 約150名 |
| 設立年 | 2015年 |
| 資本金 | 5,000万円 |
| 売上規模 | 約20億円/年 |

### FR-2: 組織構造

以下の組織構造を定義する：

```
代表取締役
├── 経営企画部（5名）
├── 管理本部（15名）
│   ├── 総務部
│   ├── 人事部
│   └── 経理部
├── 開発本部（80名）
│   ├── プロダクト開発部
│   ├── インフラ部
│   └── QA部
├── 営業本部（30名）
│   ├── 営業部
│   └── カスタマーサクセス部
└── 情報セキュリティ委員会（兼務）
    ├── ISMS管理責任者
    └── 内部監査チーム
```

### FR-3: 拠点情報

以下の拠点を定義する：

| 拠点 | 所在地 | 用途 | 従業員数 |
|------|--------|------|----------|
| 本社 | 東京都港区 | 本社機能、営業、一部開発 | 100名 |
| 開発センター | 福岡市博多区 | 開発拠点 | 40名 |
| データセンター | 東京都内（委託） | クラウドインフラ | - |
| リモートワーク | 各従業員自宅 | 在宅勤務 | 約50% |

### FR-4: 情報資産の種類

以下の情報資産を扱う：

| 分類 | 具体例 | 機密レベル |
|------|--------|------------|
| 顧客情報 | 顧客企業情報、契約情報、利用データ | 極秘 |
| 個人情報 | 顧客担当者情報、従業員情報 | 秘密 |
| 技術情報 | ソースコード、設計書、インフラ構成 | 秘密 |
| 営業情報 | 提案書、見積書、商談情報 | 社外秘 |
| 経営情報 | 財務情報、経営計画 | 極秘 |

### FR-5: 物理的セキュリティ対象

以下の物理的セキュリティ対象を定義する：

| 対象 | 説明 |
|------|------|
| オフィス入退室 | ICカードによる入退室管理 |
| サーバールーム | 本社内に小規模サーバールーム（開発環境用） |
| 執務エリア | フロアごとのゾーニング |
| 会議室 | 機密会議用の防音会議室あり |
| 受付 | 来客対応、入館証発行 |

## Non-Functional Requirements

### NFR-1: 現実性

- 実在しそうな規模・業種の組織とする
- 中小企業の IT 企業として一般的な設定とする
- 極端に特殊な設定は避ける

### NFR-2: 汎用性

- 他業種への応用が容易な設定とする
- 組織規模の拡大・縮小に対応できる構造とする

### NFR-3: 一貫性

- 全テンプレートで同じ仮想組織設定を使用する
- 部門名、役職名、拠点名は統一する

### NFR-4: 明確性

- プレースホルダーと仮想組織固有の値を明確に区別する
- 仮想組織固有の値には注釈を付ける

## Out of Scope

- 実在する企業の情報の使用
- 特定業界に特化した詳細設定
- 海外拠点の設定
- グループ会社の設定

## Dependencies

- 既存のテンプレート（プレースホルダー形式の確認）
- フォーク利用ガイド（置換手順との整合性）
